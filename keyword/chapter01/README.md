- 외래키
    
    ### 외래키
    
    한 테이블 컬럼이 다른 테이블의 기본키를 참조하도록 설정하는 연결고리
    
    외래키를 사용함으로써 존재하지 않는 유저가 생기는 것을 막을 수 있음
    
    - user 테이블에 없는 유저를 user_mission에 넣으려고 하면 에러 발생
    
    또한 함부로 삭제할 수 없도록 막음
    
    - 미션 도전 기록이 남아있는 유저를 user 테이블에서 그냥 지우려고 한다면 DB가 이 유저와 연결된 데이터가 있다며 삭제를 거부
    - CACADE 설정을 하면 같이 지울 수 있음
    
    ```sql
    ALTER TABLE `user_mission` ADD CONSTRAINT `FK_user_TO_user_mission_1` FOREIGN KEY (
    	`user_id` -- 자식 테이블의 컬럼
    )
    REFERENCES `user` (
    	`id`  -- 부모 테이블의 기본키
    );
    ```
    
    ### CASCADE 설명
    
    부모 테이블에 변화가 생겼을 때 그와 연결된 자식 데이터에도 연쇄적으로 그 변화를 전달하는 옵션을 말함
    
    - ON DELETE CASCADE (삭제 연쇄)
        - 5번 아이디를 가진 유저 삭제시 user_mission 테이블에서 user_id가 5인 미션 기록 자동으로 함께 삭제됨
    - ON UPDATE CASCADE (수정 연쇄)
        - user 테이블에서 유저 아이디를 5번→10번으로 수정하면 user_mission 테이블의 해당 user_id도 전부 자동 갱신됨
    
    ```sql
    ALTER TABLE `user_mission` 
    ADD CONSTRAINT `FK_user_TO_user_mission_1` 
    FOREIGN KEY (`id2`) REFERENCES `user` (`id`)
    ON DELETE CASCADE  -- 유저 삭제 시 미션 기록도 삭제
    ON UPDATE CASCADE; -- 유저 ID 변경 시 미션 기록의 ID도 변경
    
    ```
    
    ### But 실무에서는
    
    CASCADE 단점
    
    - 실수로 부모 데이터를 지웠을 때 수만 건의 자식 데이터가 순식간에 날아갈 수 있음
    - 부모 하나 지울 때 연결된 자식들을 일일이 찾아야 하므로 삭제 속도가 느림
    
    실무에서는 중요 데이터를 진짜 지우기 보단 deleted_at으로 소프트 delete를 더 선호함
    
- 기본키
    
    ### 기본키
    
    기본키는 각 레코드(행)를 유일하게 식별할 수 있는 아이디이다.
    
    - 중복 불가능
    - NOT NULL
    
    ```sql
    ALTER TABLE `user` ADD CONSTRAINT `PK_USER` PRIMARY KEY (`id`);
    -- 혹은 CREATE TABLE 만들 당시에 설정 가능
    ```
    
    ### 왜 기본키를 굳이 등록하지?
    
    - name + 주소 + 주민번호 칼럼을 가진 테이블에서는 굳이 기본 키가 없어도 모든 칼럼이 유니크한거 아닌가?
    - 이미 각각이 유니크한 칼럼이니깐 기본키가 필요없지 않을까?
    
    기본키를 사용해야
    
    - 데이터가 중복되지 않음을 데이터베이스 수준에서 알 수 있음
    - 기본키는 디비에서 빠르게 찾기 위한 색인(인덱스)이다.
    - 기본키는 다른 테이블과 연결하는 연결고리로 사용될 수 있다.
    - 주민번호를 외래키 사용하면 긴 글자가 똑같이 복사되어 저장되기 때문에 용량 낭비, 조인 속도 느려짐
    - 개인정보 보호 (모든 테이블에 뿌려져있으면 보안상 매우 위험하다)
    
    ### Bigint vs UUID vs UUIDv7
    
    - Bigint - 8바이트
        - 장점
            - 가장 전통적이고 성능 좋음
            - auto_increment와 함께 사용
            - 저장 공간이 작고, 인덱싱 성능 굳
        - 단점
            - 숫자가 예측 가능
            - 서버 수평 확장시 동시에 아이디를 생성할 때 충돌 날 가능성 → 중앙 관리 필요
    - UUID - 16바이트
        - 장점
            - 서로다른 서버에서 동시에 생성해도 유일함을 보장하는 128비트 식별자
            - id만 보고서 알 수 없어 보안성 good
        - 단점
            - 공간을 2배 더 차지
            - 완전 랜덤하기에 DB가 데이터 저장할 때 위치 찾느라 성능 떨어짐 (인덱스 단편화 현상)
    - UUID v7 - 16바이트
        - 장점
            - uuid의 유일성과 bigint의 시간 순서를 합친 최신 표준 방식
            - 앞에 타임스탬프를 넣고, 뒷부분에 랜덤 값을 넣음
            - 정렬 가능하기 때문에 디비 인덱스 성능이 bigint만큼 좋음
            - 분산 환경에도 서버 간 겹치지 않고 시간 순서까지 유지됨
    
- ER 다이어그램
    
    ### ERD
    
    데이터베이스 만들기 전, 서비스에 어떤 엔티티가 있고 어떤 관계를 맺는지 그림으로 그려낸 설계도
    
    요소
    
    - 엔티티 - 관리할 대상
    - 속성 - 대상이 가진 정보
    - 관계 - 대상 사이의 연결
    
    ### 식별관계 vs 비식별관계
    
    ERDcloud.com에서 erd 작성 시에도 관계설정할 때 식별관계와 비식별관계를 선택할 때 많이 고민한다.
    
    - 식별관계 : 자식 엔티티가 부모의 PK를 자신의 PK 일부로 사용 (파란 실선)
    - 비식별관계 : 자식이 부모의 아이디를 FK로 일반 칼럼으로 사용 (빨간 점선)
    
    ### 실무에서는 비식별 관계를 선호
    
    - 식별관계를 PK가 복잡해지고 유연성 떨어짐
        - 부모의 PK를 자신의 PK로 가져오기 때문에 계층이 깊어질수록 자식의 PK 개수 늘어남
        - 반면 비식별관계는 자신만의 단일 PK를 갖고, 부모 아이디는 그냥 외래키 컬럼으로만 존재
    - 비즈니스 변화에 유용한 비식별관계
        - 실무에서 부모 데이터 바뀌거나 관계 끊어지는 일 빈번하다
        - 만약 식별관계에서 부모 아이디가 바뀐다면?
            - ex) 사장님과 가게가 1대N 관계를 갖는다. 이때 사장님의 아이디가 10에서 20으로 바뀐다면 가게의 PK도 (1, 10)에서 (1, 20)으로 바뀐다.
            - 또한 이 가게를 참조하는 테이블들, 미션이나 리뷰가 있다면 그곳도 전부 20으로 업데이트해야함
- 복합 키
    
    ### 복합키
    
    하나의 컬럼만으로는 데이터 중복 막을 수 없을 때, 두 개 이상 컬럼 합쳐서 하나의 기본키 만드는 것
    
    ![image.png](attachment:64848ede-1d5b-4339-a30b-2374b8063991:image.png)
    
    ```sql
    ALTER TABLE `user_mission` ADD CONSTRAINT `PK_USER_MISSION` PRIMARY KEY (
    	`id`,       -- 미션 ID
    	`store_id`, -- 가게 ID
    	`id2`       -- 유저 ID
    );
    
    ```
    
    위와 같이 이렇게 SQL 코드로 3개 칼럼을 묶어서 하나의 기본키로 사용할 수도 있다.
    
    보통 식별관계를 사용하면 자동으로 위와같이 복합키가 된다.
    
    ![image.png](attachment:3ab30720-b684-4eb1-9214-672d798df33f:image.png)
    
    하지만 user_mission와 1대1로 연결되는 리뷰 테이블에서도 이렇게 무겁게
    
    (미션아이디, 음식점아이디, 유저아이디) 를 복합키로 포함하게 되어 무거워진다.
    
    ### 실무적 고민 :  가볍게 할 방법은 없을까?
    
    유저미션 테이블에서 복합키를 버리고, 간단한 PK 하나를 만들고,
    
    대신 중복 방지를 위해 id, store_id, id2에는 유니크 제약을 건다. (id, store_id, id2)에 PK 복합키로 유일성을 가지는 것과 동일한 효과이다.
    
    이후, Review 테이블에서는 위처럼 3개 복합키를 가져올 필요 없이 부모의 가벼운 id 하나만 외래키로 가져오면 된다.
    
- 연관관계
    
    ### 일대다
    
    한쪽 데이터가 반대 데이터 여러개와 연결되는 구조
    
    예시로는 유저와 리뷰 테이블
    
    - 유저 1명은 여러개의 리뷰를 쓸 수 있지만, 리뷰 하나는 반드시 작성자 하나에 속해야한다.
    - 이때 N쪽, 즉 리뷰 테이블에 유저의 PK를 외래키로 심어야한다.
    
    ### 다대다
    
    양쪽 모두가 서로 여러 개와 연결될 수 있는 구조이다.
    
    예를 들어 유저와 미션
    
    - 유저 1명은 여러 미션에 도전할 수 있다.
    - 미션 1개도 여러 명의 유저가 도전할 수 있다.
    - 두 테이블 사이에 매핑테이블을 하나 더 만든다. user_mission. 중간 테이블에는 유저아이디와 미션아이디 쌍을 보관한다.
    
    ### 일대일
    
    데이터 하나가 딱 하나의 데이터하고만 매칭되는 구조로,
    
    유저 테이블의 컬럼이 많아질 때 user_privacy_info 테이블로 분리하는 것을 예로 들 수 있다.
    
    - 한쪽의 PK를 다른 쪽의 PK이자 FK로 사용한다.
- 정규화
    
    [정보처리 실기_데이터베이스06강_정규화](https://youtu.be/RXQ1kZ_JHqg?si=f0OPsoOWnJXSbqca)
    
    ### 0. 정규화전(비정규형)
    
    하나의 테이블에 모든 정보 때려넣은 상태
    
    ### 1. 제1정규화
    
    한 칸에는 무조건 값 하나만, 즉 모든 칼럼은 원자값을 가져야한다.
    
    예를 들어 img_url 컬럼에 사진1, 사진2, 사진3 이렇게 3장이 쉼표로 구분해서 넣어져있다면 위반.
    
    따로 Image 테이블로 빼거나 행을 나누어 저장한다.
    
    ### 2. 제2정규화
    
    부분함수종속을 분리한다 - 기본키 전체에만 의존하기
    
    review 테이블에 mission_id, store_id, user_id, nickname 이렇게 있을 때
    
    username은 user_id를 통해서 알 수 있으므로 user 테이블로 분리하는 것이 좋다.
    
    ### 3. 제3정규화
    
    이행적 종속 끊기
    
    가게 아이디 → 카테고리 아이디 → 카테고리 이름
    
    이때 가게 아이디를 알면 카테고리 아이디를 알고, 카테고리 아이디를 알면 이름을 알게되므로,
    
    카테고리 이름이 중복 저장되지 않게 food_category 테이블을 따로 둔다.
    
- 반 정규화
    
    ### 반 정규화
    
    정규화를 통해 잘 쪼개놓은 테이블을 성능 향상을 위해 의도적으로 다시 합치거나 중복 데이터를 허용
    
    → 서비스에서 Join 연산을 줄여 응답 시간을 단축하기 위해
    
    1. 칼럼 중복
        - review 보여줄 때 작성자 닉네임 필요한데, user 테이블과 조인을 하지 않고 review 테이블에 user_nickname을 칼럼을 추가해버리면 join하지 않고도 한 테이블만 읽어도 됨
        - DB 상에서는 유저의 닉네임이 review 테이블에도 있고 유저 테이블에도 있게 되지만 서비스상에서 성능 굳
    2. 계산/통계 값 저장
        - like_count, comment_count, mission_success_count 와 같이 칼럼을 두어 특정 이벤트 발생 시 +1 함으로써 통계값 조회 시 조인하지 않고도 빠르게 값을 읽을 수 있다.
        - 다만, 서비스 함수 내에서 별도 로직을 코딩해야한다.
    
    ### 반정규화 유의점
    
    - 데이터 불일치
        - 위의 예시에서 유저가 닉네임 바꿨는데 review 테이블에 복사해둔 닉네임을 바꾸지 않았다면 데이터 불일치 발생함
